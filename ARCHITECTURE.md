# Architecture Guide

> For AI agents and contributors working with this codebase.

## Overview

**arch-autoinstall** builds a customized Arch Linux ISO that performs a
fully automated installation. The user's configuration (disk layout, desktop,
encryption, credentials) is baked into the ISO at build time so that booting
the ISO triggers an unattended install.

The project is a single ~3000-line Bash script (`build-iso.sh`) plus a handful
of helper scripts in `scripts/`. There are no external frameworks, Python
packages, or package-manager abstractions — just Bash, `dialog` for the TUI,
and standard Arch Linux tools.

## Build Pipeline

`build-iso.sh` runs this pipeline:

```
1. Parse CLI args (--preferred, --creds, --config)
2. Load credentials JSON (if --creds)
3. Run TUI wizard (or apply_preferred() if --preferred)
4. Show configuration review → confirm
5. Download latest Arch ISO from OSUOSL mirror (cached)
6. Extract ISO → extract airootfs.sfs (squashfs)
7. Inject into squashfs:
   ├── /root/arch-autoinstall/config/       ← archinstall JSON configs
   ├── /root/arch-autoinstall/scripts/      ← helper scripts
   ├── /root/arch-autoinstall/autorun.sh    ← generated install script
   ├── /root/arch-autoinstall/post-install.sh ← generated post-install
   └── /etc/profile.d/99-autoinstall.sh     ← auto-launch hook
8. Repack squashfs → regenerate checksums → build ISO with xorriso
```

Steps 6–8 require `sudo` (squashfs extraction/repacking runs as root).

## Generated Scripts

`build-iso.sh` generates two scripts from heredocs. These are **not** static
files — they're templated with `__PLACEHOLDER__` values that get replaced
via `sed` after the heredoc is written.

### autorun.sh (runs inside live ISO)

Generated by `generate_autorun()` (~500 lines). This is the main install
orchestrator that runs when the live ISO boots. Flow:

```
1. Configure proxy (if enabled) — must run before network check
2. Wait for network connectivity (ping or curl through proxy)
3. Detect target disk (auto or interactive)
4. Partition disk (alongside-Windows or standard layout)
5. Prompt for LUKS password (if not baked in)
6. Prompt for user password (if not baked in)
7. Call run_archinstall() — tries bundled version, falls back to ALA
8. Re-mount installed root filesystem
9. Copy scripts + post-install to installed system
10. Apply proxy to installed system via arch-chroot (if enabled)
11. Prompt for reboot
```

Key placeholders: `__AUTO_DISK__`, `__DISK_LAYOUT__`, `__ENABLE_LUKS__`,
`__ENABLE_PROXY__`, `__PROXY_URL__`, `__ARCHINSTALL_FALLBACK_VER__`

### post-install.sh (runs on first boot of installed system)

Generated by `generate_post_install()` (~200 lines). Run manually after
rebooting into the installed system. Handles:

- Hibernate setup (calls `enable_hibernate_swapfile.sh`)
- Secure Boot key creation (calls `setup-secureboot.sh`)
- TPM2 enrollment (calls `setup-tpm-unlock.sh`)
- illogical-impulse clone + setup + feature branch deployment
- Bluetooth/CUPS service enablement

Key placeholders: `__ENABLE_HIBERNATE__`, `__ENABLE_TPM__`, `__ENABLE_II__`,
`__ENABLE_II_FEATURES__`, `__II_SELECTED_BRANCHES__`

## archinstall Integration

### Config Format (archinstall 4.x)

The archinstall JSON config is generated by `generate_archinstall_config()`.
Critical details for anyone modifying this:

- **`app_config`** is the top-level key for audio/bluetooth/printing:
  ```json
  "app_config": {
      "audio_config": {"audio": "pipewire"},
      "bluetooth_config": {"enabled": true},
      "print_service_config": {"enabled": true}
  }
  ```
- **Profile names are case-sensitive**: `"GNOME"` not `"Gnome"`, `"Hyprland"` is correct
- **`encryption_password`** is a top-level key (not inside `disk_encryption`)
- **`--silent`** flag is required for unattended install (skips TUI)
- **Credentials file** (`--creds`) is a separate JSON with `!users` array
- **`Password("")`** raises `ValueError` — empty `!password` causes user to be skipped

### Version Fallback

`run_archinstall()` (defined inside the autorun heredoc) handles archinstall
version instability:

1. Try the bundled archinstall version from the ISO
2. On failure, download a known-good version from the
   [Arch Linux Archive](https://archive.archlinux.org/packages/a/archinstall/)
3. Install it with `pacman -U --noconfirm` and retry

The fallback version is controlled by `ARCHINSTALL_FALLBACK_VER` (default: `4.3-1`).

### Post-archinstall Remount

archinstall unmounts everything when it completes in `--silent` mode. The
autorun script re-mounts the installed root filesystem (btrfs `@` subvolume)
at `/mnt/archinstall` to copy post-install scripts and apply proxy config.
Detection order: open LUKS mapper → btrfs on target disk → any btrfs device.

## Disk Layouts

Three supported layouts:

| Layout | Partitions | Use Case |
|--------|-----------|----------|
| `standard` | ESP+root (2 partitions) | Clean single-boot install |
| `dual-clean` | ESP+XBOOTLDR+root (3 partitions) | Wipe disk, prep for dual-boot |
| `alongside` | Reuse Windows ESP, create XBOOTLDR+root in free space | Install next to Windows |

All layouts use btrfs with subvolumes: `@` (root), `@home`, `@log`, `@pkg`,
`@.snapshots`, `@swap`.

## Configuration Variables

All config state lives in global shell variables (lines 45–160 of `build-iso.sh`).
Key groups:

| Group | Variables | Notes |
|-------|----------|-------|
| Desktop | `ENABLE_HYPRLAND`, `ENABLE_GNOME`, `ENABLE_KDE`, etc. | Multiple can be enabled |
| Security | `ENABLE_LUKS`, `ENABLE_HIBERNATE`, `ENABLE_TPM` | LUKS required for hibernate+TPM |
| User | `USERNAME_CFG`, `USER_PASSWORD`, `LUKS_PASSWORD` | Set via TUI, --creds, or prompted |
| Proxy | `ENABLE_PROXY`, `PROXY_URL` | Corporate network support |
| Disk | `DISK_LAYOUT`, `TARGET_DISK`, `AUTO_DISK` | Layout determines partition scheme |
| Power | `SUSPEND_MODE`, `SLEEP_ACTION`, `HIBERNATE_DELAY`, etc. | Sleep/hibernate behavior |
| Pinning | `ARCHINSTALL_FALLBACK_VER` | Known-good archinstall version |

## Testing

```bash
# Syntax check all scripts
make check

# Build with preferred config
make preferred

# Or with credentials
./build-iso.sh --preferred --creds configs/credentials.json

# Launch QEMU VM
./test-vm.sh out/arch-autoinstall-*.iso

# Boot from disk (no ISO) after install
qemu-system-x86_64 -enable-kvm -machine q35 -cpu host -smp 4 -m 4096 \
    -drive if=pflash,format=raw,readonly=on,file=/usr/share/edk2/x64/OVMF_CODE.4m.fd \
    -drive if=pflash,format=raw,file=vm/OVMF_VARS.4m.fd \
    -drive file=vm/disk.qcow2,format=qcow2,if=virtio \
    -vga virtio -display gtk
```

### QEMU Notes

- `test-vm.sh` creates a 60G qcow2 disk and copies fresh OVMF UEFI vars
- OVMF_VARS stores UEFI boot entries — reset it to force ISO boot
- The virbr0 bridge provides NAT networking (192.168.122.0/24)
- LUKS password, user password: whatever was in your `credentials.json`

## Credentials JSON Format

```json
{
    "username": "myuser",
    "user_password": "secret",
    "luks_password": "secret"
}
```

Loaded with `--creds configs/credentials.json`. Supports GPG-encrypted
files (`.gpg` extension — decrypted at load time).

## Common Pitfalls

1. **Never use `json.dump` to patch squashfs configs** — it converts bare
   integer placeholders to strings. Use `sed` for placeholder replacement.
2. **`set -euo pipefail`** is active — `(( expr ))` returning 0 as false
   will exit the script. Guard with `|| true` where needed.
3. **The squashfs is extracted as root** — all `sudo` commands in the
   extraction/repacking phase operate on `$work/squashfs_extract/`.
4. **archinstall 4.x changed significantly** from 2.x — `disk_encryption`
   format, profile names, `app_config` structure, and `--silent` flag are
   all different. Always test against the current archinstall version.
5. **The autorun heredoc uses `'AUTOEOF'`** (single-quoted) so variables
   aren't expanded — all runtime values use `__PLACEHOLDER__` + `sed`.

## Branch Strategy

- **`master`** — Public release. Generic autoinstaller.
- Feature branches for site-specific customizations (e.g. corporate proxy)
  can be rebased on master.

When making changes:
- Generic fixes → commit on `master`
- Site-specific changes → commit on feature branches, rebase on master
